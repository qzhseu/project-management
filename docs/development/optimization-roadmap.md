## 项目管理助手 — 通盘审查与优化计划

---

### 一、项目现状总览

本项目是一个面向项目管理场景的桌面端AI助手，技术栈为 Electron 42 + React 19 + TypeScript + Vite 7 + Ant Design 6 + Tailwind CSS 4，后端使用 sql.js（SQLite）做本地数据持久化，集成了智谱AI和小米MiMo两个大模型提供商。项目经历了从 Tauri 到 Electron 的整体迁移，目前前端已全面接入 Electron 后端，Tauri Rust 后端为遗留代码。

**已实现的核心功能**：项目 CRUD、文件上传与拖拽、文件内容提取（txt/md/pdf/docx/xlsx）、AI 自动分类（按阶段/按内容两种模式）、AI 对话（含文件上下文选择）、对话历史持久化、设置页（AI 配置 + 提取配置 + 提示词模板）、面包屑导航、项目目录结构自动创建。

**总体评估**：功能骨架搭建较为完整，但多处功能停留在"能跑"层面，缺乏打磨。UI 视觉质量偏低，大量使用 Ant Design 默认样式，没有形成统一的设计语言。代码层面存在较多遗留的死代码和双后端维护负担。

#### 2.4 代码质量问题

**安全问题**：`src-tauri/src/config.rs` 中硬编码了智谱 API Key（`sk-clwzd...`），已提交到代码库。虽然是遗留的 Tauri 代码，但存在安全隐患。

### 三、UI 问题诊断

#### 3.1 核心视觉问题

**整体观感偏"后台管理系统"**：大量使用 Ant Design 默认组件样式（蓝色主按钮、默认表格、标准 Modal），看起来像是一个通用的管理后台，而非面向终端用户的项目管理工具。缺乏品牌感和个性。

**布局浪费空间**：`max-w-6xl mx-auto`（最大宽度 1152px）在大屏上两侧留白过多，而项目详情页的左侧导航 + 右侧内容区域在小窗口上又显得拥挤。没有充分利用桌面应用的窗口空间。

**拖拽区域粗糙**：文件上传区域使用了内联 style（`border: 2px dashed #d9d9d9`、`backgroundColor: #fafafa`），是纯功能实现，没有任何视觉设计。与 Notion/Linear 等标杆产品的精致拖拽区域差距很大。

**缺少空状态设计**：项目列表为空、文件列表为空、对话历史为空时，都没有专门的空状态展示（插图 + 引导文案），用户体验不完整。

**色彩体系零散**：阶段标签的颜色硬编码在组件内（`STAGE_STYLE` 对象），且多处重复定义（ProjectList 和 ProjectHome 各有一套）。没有使用 CSS 变量或 Tailwind 配置来统一管理色彩。

**无动效和过渡**：页面切换是瞬间跳转（无动画），文件上传/分类等操作的进度反馈只有 loading 状态和 message 弹窗，缺少平滑的状态过渡。

#### 3.2 交互体验问题

**项目列表只有表格视图**：对于项目管理工具，卡片视图（Card View）通常更直观，可以展示更多信息（描述、标签、进度）。当前只有 Table 一种展现形式。

**对话页上下文面板不够直观**：左侧的文件选择是简单的 checkbox 列表，没有展示文件类型图标、分类标签或内容预览，用户难以判断该选择哪些文件作为对话上下文。

**分类结果缺少反馈环节**：AI 自动分类后只弹一个 `message.success`，用户无法在上传流程中确认或修改分类结果。设计文档明确要求"无确认步骤"，但手动重新分类的入口也没有做。

**设置页缺乏引导**：三个 Tab 直接堆叠表单字段，没有分组说明、帮助文本或配置校验提示，对不熟悉 AI 配置的用户不够友好。

以下按优先级从高到低排列，分为三个批次。

#### 批次一：基础修复与代码清理（建议 3-5 天）

**目标**：消除安全隐患、清理死代码、修复已知 Bug、补齐缺失的类型声明。

**4.1 安全修复**

- 如果 Tauri 后端确定不再使用，考虑将整个 `src-tauri/` 目录归档或移除

**4.2 死代码清理**
- 删除或归档以下文件：
  - `src/components/FileManager/` 整个目录（4个文件）
  - `src/components/ProjectList/ProjectTable.tsx`
  - `src/components/Chat/UploadZone.tsx`
  - `src/components/Settings/PromptEditor.tsx`（功能应合并到 SettingsPage）
  - `src/mocks/demoProjects.ts`
- 清理 `src/types/index.ts` 中被遗留组件引用的未使用类型（如果确认不再需要）

**4.3 Bug 修复**

#### 批次二：UI 重设计与体验提升（建议 7-10 天）

**目标**：建立统一的设计语言，大幅提升视觉质量和交互体验，向 Notion/Linear 等标杆产品看齐。

**4.5 设计系统建立**

建立统一的设计 Token 体系，使用 Tailwind CSS 配置统一管理：

- **色彩**：定义品牌色（主色、辅色）、语义色（成功/警告/错误/信息）、中性色梯度，通过 Tailwind 的 `theme.extend.colors` 配置。移除组件内硬编码的颜色值。
- **间距与圆角**：统一 spacing scale（4px 基准），统一 border-radius（sm: 6px, md: 8px, lg: 12px, xl: 16px）。
- **字体层级**：定义 display/heading/body/caption 等字体大小和行高，使用 `font-feature-settings` 优化中文排版。
- **阴影层级**：定义 elevation-1 到 elevation-4 的阴影系统，替代 Ant Design 默认的 box-shadow。
- **动效曲线**：定义统一的 transition 预设（ease-out、spring 等），所有交互元素统一使用。

**4.6 布局重构**

- **项目列表页**：改为全宽布局（移除 `max-w-6xl` 限制），利用窗口全部空间。增加"卡片视图/表格视图"切换按钮。卡片视图每个项目展示项目名称、状态标签、文件数量、最近更新时间和分类方式标签。
- **项目详情页**：左侧导航改为可折叠的侧边栏（默认展开，可收起为图标模式），内容区自适应宽度。文件区域改为"上方拖拽区 + 下方文件列表"的紧凑布局。
- **对话页**：全屏沉浸式布局，上下文文件面板改为右侧可折叠抽屉（默认收起），对话区域最大化。
- **移除 App.tsx 的外层 padding 和 max-width 限制**，让每个页面自行控制布局。

**4.7 组件级 UI 优化**

**项目列表页**：
- 空状态：展示一个插图 + "创建你的第一个项目" 引导
- 卡片视图：带渐变边框的项目卡片，hover 时有微妙的阴影提升
- 搜索和筛选栏：改为更紧凑的内联筛选，使用 Segmented 组件切换排序方式
- 新建项目按钮：使用更突出的样式，或在空状态时作为 CTA 大按钮

**项目详情页（ProjectHome）**：
- 拖拽区域重新设计：使用渐变虚线边框、图标动画（拖拽时图标放大 + 颜色变化）、支持的文件格式图标展示
- 文件列表表格：减小行高、增加文件类型图标列、分类标签使用更柔和的配色、操作按钮改为 hover 时显示的工具栏
- StageNav 重新设计：每个阶段项带文件数量 badge、当前选中项有左侧彩色指示条、支持拖拽排序（自定义阶段时）

**对话页（ChatWindow）**：
- 消息气泡重新设计：用户消息右对齐使用品牌色背景，AI 消息左对齐使用中性背景
- 流式响应：实现逐字打字效果（打字机动画），配合"正在思考"骨架屏
- 上下文文件面板：文件项带类型图标和分类标签，选中状态有视觉高亮
- 输入区域：增大输入框高度，支持 Markdown 预览，发送按钮带微动画

**设置页（SettingsPage）**：
- 分组卡片布局：每个配置区域用卡片包裹，带标题和简要说明
- AI 配置区：提供商切换时使用卡片选择器而非下拉框，API Key 输入带可见性切换
- 提示词编辑器：集成 PromptEditor 组件，支持变量插入和实时预览

**4.8 动效与过渡**

- 页面切换动画：使用 `framer-motion` 实现淡入淡出 + 轻微位移的过渡效果
- 列表项动画：新增/删除项目或文件时，使用 layout animation 平滑过渡
- 按钮反馈：所有可点击元素 hover/active 状态有明确的视觉变化
- 加载状态：用骨架屏（Skeleton）替代 Ant Design 的 Spin 组件，更自然
- 拖拽反馈：文件拖入时拖拽区域有明显的视觉变化（边框变色、图标放大、背景色变化）

**4.9 空状态与异常状态**

为所有列表和空内容区域设计空状态：
- 项目列表为空：插图 + "还没有项目，立即创建" + CTA 按钮
- 文件列表为空：插图 + "拖拽文件到此处开始管理" 引导
- 对话历史为空：插图 + "向 AI 助手提问，开始对话" 引导
- 网络错误/AI 调用失败：友好的错误提示卡片，带重试按钮和配置检查引导

### 六、技术决策建议

**关于 Tauri 后端**：建议将 `src-tauri/` 目录重命名为 `src-tauri-archived/` 或直接删除（保留在 git 历史中）。双后端并存增加了新开发者理解项目的成本，也容易产生混淆。

**关于 Ant Design 的使用策略**：Ant Design 作为基础组件库保留，但需要在以下方面做定制化处理——通过 ConfigProvider 全局定制主题（色彩、圆角、字体），对 Table、Modal、Button 等高频组件做二次封装，逐步建立项目自己的组件层。不建议完全替换 Ant Design（迁移成本太高），而是"在 Ant Design 之上建立自己的设计语言"。

**关于状态管理**：当前使用 useState 管理全局状态已经有些吃力（页面切换、项目选择等都是 prop drilling）。建议引入 Zustand 做轻量级状态管理，替代当前的 useState 模式，为后续功能扩展（如多项目同时打开、全局通知等）做准备。

**关于路由**：建议从 useState 页面切换迁移到 React Router（`react-router-dom`），支持浏览器前进/后退、深度链接、以及未来可能的多窗口支持。Electron 中可以使用 hash router。

### 七、执行建议

建议按"批次一 → 批次二前半段 → 批次三核心功能 → 批次二后半段 → 批次三工程化"的顺序穿插执行。具体来说：先做安全修复和代码清理（批次一），然后立刻开始设计系统建立和布局重构（批次二前半段），这样后续的功能开发可以在新的 UI 框架上进行，避免返工。核心功能（自定义阶段、手动分类）放在 UI 框架稳定后实现。动效和打磨放在最后做锦上添花。
