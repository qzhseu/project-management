# 签字检测方案分析

> 创建时间：2026-06-11

---

## 背景

上传PDF或图片格式的签字文件时，检测是否有手写签字。有签字=关键签字件，无签字=待签字打印版本。

核心流程：**PDF/图片 → 生成图片 → 多模态AI判断"是否有手写签字"**

差异点在于：**PDF如何转成图片送AI分析**

---

## 方案对比

### 方案A：pdfjs-dist 的 getOperatorList()

**原理**：
- pdfjs-dist 是 Mozilla 开发的 PDF 解析库
- `getOperatorList()` 可以提取 PDF 页面的绘制指令（画线、填色、文字等）
- 通过分析绘制指令中的"深色连通区域"来判断是否有手写笔迹

**实现方式**：
```
PDF文件 → pdfjs-dist解析 → 提取绘制指令 → 分析笔画密度 → 判断是否有签字
```

**优势**：
- 不需要额外依赖（项目已有pdfjs-dist）
- 不需要创建额外的BrowserWindow
- 可以在主进程运行
- 处理速度快

**劣势**：
- 实现复杂：需要自己编写笔画分析算法
- 准确率有限：只能分析矢量指令，扫描件（图片PDF）无法处理
- 需要区分"印刷文字"和"手写笔迹"

**准确率预估**：60-70%（矢量PDF）/ 无法处理（扫描件）

**工作量**：2-3天

---

### 方案B：Electron printToPDF（反向）+ 暴露 BrowserWindow

**原理**：
- 创建一个隐藏的 BrowserWindow
- 用 pdfjs-dist 将PDF渲染到隐藏窗口的Canvas上
- 用 `nativeImage` 截取Canvas内容生成图片
- 将图片发送给多模态AI分析

**实现方式**：
```
PDF文件 → 创建隐藏BrowserWindow → pdfjs-dist渲染到Canvas → nativeImage截图 → 多模态AI判断
```

**优势**：
- 准确率高：渲染效果与用户看到的一致
- 支持所有PDF类型（矢量+扫描件）
- 实现相对简单

**劣势**：
- 需要创建隐藏BrowserWindow，有资源开销
- 渲染过程在渲染进程，需要IPC通信
- 首次加载pdfjs-dist的worker可能有延迟

**准确率预估**：85-95%

**工作量**：1-2天

---

### 方案C：pdf2pic 库

**原理**：
- pdf2pic 是一个 Node.js 库，底层使用 GraphicsMagick/ImageMagick
- 直接将PDF页面转换为PNG/JPEG图片

**实现方式**：
```
PDF文件 → pdf2pic转换 → PNG图片 → 多模态AI判断
```

**优势**：
- 使用简单，API简洁
- 转换质量高
- 支持多种输出格式

**劣势**：
- 依赖外部工具（GraphicsMagick/ImageMagick）需要安装
- Electron打包时需要打包这些外部工具
- 增加安装包体积（约20-30MB）
- 跨平台兼容性问题

**准确率预估**：85-95%（取决于图片质量）

**工作量**：1天（但打包配置复杂）

---

## 推荐方案

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 准确率 | 60-70% | 85-95% | 85-95% |
| 工作量 | 2-3天 | 1-2天 | 1天 |
| 依赖复杂度 | 低 | 低 | 高 |
| 打包体积 | 无增加 | 无增加 | +20-30MB |
| 跨平台 | 好 | 好 | 有风险 |
| 推荐度 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |

**建议选择方案B**：
- 准确率最高
- 无需额外依赖
- 打包体积无影响
- 实现难度适中

**方案A的问题**：分析矢量指令判断手写笔迹算法复杂，且无法处理扫描件PDF，实际效果可能很差。

**方案C的问题**：依赖外部工具，Electron打包复杂度高，不推荐。
