## 二、UI 设计审查要点

### 2.1 布局结构审查

请审查以下布局是否存在遮挡、溢出、对齐问题：

#### App Header（App.tsx:114-168）
- 项目列表页：Logo + "项目管理助手" | 面包屑 + 设置按钮 + 样式验证按钮
- 项目详情页：返回箭头 + 项目名 + 阶段 | 面包屑 + 对话按钮 + 设置按钮

#### 项目卡片（ProjectList.tsx:248-425）
- 顶部 4px 彩色条 → 标题行（名称 + 状态标签）→ 元信息 → 时间线 → 底部（时间 + 操作按钮）
- **审查点**：卡片内各元素间距是否一致？hover 效果是否流畅？

#### 统计卡片（ProjectStats.tsx）
- 4 个等宽卡片：文件数、关键问题、待处理、AI 摘要
- **审查点**：第 4 个卡片是否在 grid 内正确对齐？

#### 阶段侧边栏（StageSidebar.tsx）
- 200px 宽度，11 个分类阶段 + 所有文件 + 未分类
- **审查点**：11 个阶段在 200px 下是否需要滚动？每个阶段项的图标+文字+计数是否对齐？

#### 对话窗口（ChatWindow.tsx）
- 左侧会话列表（48px 折叠/280px 展开）+ 主对话区
- **审查点**：消息气泡最大宽度 70% 是否合理？输入区域高度自适应是否正常？

## 三、代码质量审查要点

### 3.1 类型安全

**审查点**：
- `src/types/windowApi.ts` 中的类型声明是否与 `electron/preload.ts` 的实现参数一致？
- 是否存在 `any` 类型、`as unknown` 强转、非必要的 `!` 非空断言？
- `src/types/index.ts` 中的接口定义是否完整？

**已知问题**：
- `windowApi.ts:44`：`getModelList` 返回类型 `data?: any[]` → 应改为 `AIProviderConfig[]`
- `src/__tests__/utils.test.ts`：测试内部定义函数而非测试产品代码

### 3.2 错误处理

**审查点**：
- 所有 IPC handler 是否有顶层 try/catch？
- 错误返回格式是否统一（`{success: false, error: string}`）？
- AI 请求失败时用户是否有明确反馈？
- 是否有未捕获的 Promise rejection？

**已知问题**：
- `ai:classify` 和 `ai:analyze` 缺顶层 try/catch（部分 handler 裸 reject）
- 多处空 `catch {}` 块吞掉错误

### 3.3 安全性

**审查点**：
- 所有文件操作是否有路径安全校验？
- API Key 是否加密存储（safeStorage）？
- CSP 策略是否生效？
- BrowserWindow 是否启用 sandbox？
- 是否有路径遍历风险？

### 3.4 性能

**审查点**：
- 大列表是否有虚拟滚动？
- React 组件是否合理使用 memo/useMemo/useCallback？
- 数据库写入是否有并发保护？
- 文件上传是否有分片处理？

### 3.5 一致性

**审查点**：
- `windowApi.ts` 类型 vs `preload.ts` 实现：参数个数、类型是否一致？
- 前端 `aiService.ts` 调用 vs 后端 `ai-handlers.ts` 处理：参数是否匹配？
- 分类 Prompt 中的阶段列表 vs `STAGE_STYLE` 中的阶段列表：是否一致？
